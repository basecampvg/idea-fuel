Storage Architecture: Where do the images live?

Each approach affects cost, complexity, and how the AI pipeline accesses images

A

Supabase Storage (Recommended)

You're already on Supabase for Postgres. Their Storage service gives you S3-compatible object storage with built-in auth policies, presigned URLs, and image transforms — zero new infrastructure.

Pros

  • No new accounts or credentials — same Supabase project
  • Row-level security policies match your existing auth
  • Built-in image transforms (thumbnails, resize)
  • Presigned upload URLs — mobile uploads direct to storage, no server proxy
  • Free tier: 1GB storage, 2GB bandwidth

Cons

  • Vendor lock-in to Supabase (though S3-compatible API)
  • Adds @supabase/supabase-js dependency to server
B

Cloudflare R2

S3-compatible storage with zero egress fees. Great for high-read workloads, but requires a new Cloudflare account and separate auth layer.

Pros

  • No egress fees (reads are free)
  • S3-compatible API
  • 10GB free tier

Cons

  • New account + credentials to manage
  • Must build your own auth/access layer
  • No built-in image transforms
C

Direct-to-DB (Base64 in JSONB)

Store compressed images directly in Postgres as base64 in a JSONB column. Simplest implementation — no external storage service at all.

Pros

  • Zero infrastructure — just a column
  • Atomic with project creation (no orphaned files)
  • Simplest implementation by far

Cons

  • Bloats database size (3-5 images × ~500KB = 1.5-2.5MB per idea)
  • Slower queries when selecting projects
  • Supabase DB has storage limits per plan
  • Not scalable if you add more attachment types later
